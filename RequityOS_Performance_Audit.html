Executive Summary

A deep audit of RequityOS identified 16 performance bottlenecks across the pipeline, CRM, and core platform. The slowdown you are feeling is real and has specific, fixable causes. The two biggest offenders are aggressive cache invalidation on inline saves (every field edit re-renders the entire pipeline) and loading all contacts/companies without pagination (the DOM grows linearly with your data).

The good news: the architecture is solid. Parallel queries, React Server Components, and optimistic updates are already in place in many areas. The issues are localized and surgical. A focused sprint of 8-10 dev days can deliver a 2-4x perceived speed improvement on the pages you use most.

Current State Snapshot

  • Pipeline deal detail: 20+ sequential Supabase queries for commercial deals across 3 phases
  • Pipeline inline save: Every field edit invalidates /pipeline cache, triggering full re-render of all deal cards
  • Contacts list: Loads ALL contacts into DOM. No pagination, no virtualization, no search debouncing
  • Contact 360: 16 parallel queries (good) + waterfall field config query (bad) + all tab data upfront (wasteful)
  • Borrower portal: Polls for messages every 10 seconds even when tab is hidden
  • Every page load: Middleware and layout both query profiles table independently (double DB round trip)

Detailed Breakdown by Area

Pipeline (Kanban + Deal Detail)

The pipeline is the most performance-sensitive area of the portal. Luis and Estefania live in these views all day. Two issues dominate:

  • revalidateDeal after inline saves: In pipeline/[id]/actions.ts, revalidateDeal() is called after logging activity, quick actions, sheet URL changes, and team member assignments. Each call invalidates both /pipeline and /pipeline/[id], forcing Next.js to re-fetch and re-render the entire pipeline. The fix is to remove revalidateDeal from inline save paths and rely on optimistic local state updates only.
  • Sequential query phases on deal detail: The deal detail page.tsx runs queries in 3 sequential phases. Phase 1 fetches the deal. Phase 2 runs 12 queries in Promise.all (good). Phase 3 conditionally fetches 8 more commercial UW queries, but only AFTER Phase 2 resolves. These should all be parallelized into a single Promise.all with conditional inclusion.
  • Redundant field config hooks: useUwFieldConfigs, useDealLayout, and useFieldConfigurations each fire independent Supabase queries when their components mount. On a deal detail page, the same field configuration data is fetched 3-5 times. A single fetch at the page root, passed via context, would eliminate this.
  • PipelineKanban formula map: The formula map in PipelineKanban.tsx is rebuilt every render without memoization. This object is passed to every DealCard, so reference changes trigger re-renders of all visible cards.

CRM (Contacts + Companies)

The CRM is where the scaling pain will hit hardest. Every query loads the full dataset, and every interaction filters in-memory.

  • No pagination: contacts/page.tsx and companies/page.tsx both load the entire table with no .range() or .limit(). At 500+ contacts, this means fetching and rendering all rows. Cursor-based pagination with a page size of 50 is the standard fix.
  • No search debouncing: contacts-view.tsx and companies-view.tsx re-run the full filter chain on every keystroke. A 300ms debounce on the search input would eliminate the jank.
  • No list virtualization: Every filtered contact renders as a full DOM row with TooltipProvider, Badge, Link, and Avatar components. At 200+ rows, this causes visible layout thrashing. react-virtual would render only visible rows (~15-20 at a time).
  • Contact 360 loads all tabs upfront: When opening a contact, the page fetches activities, emails, tasks, loans, and pipeline data all at once, even though only the overview tab is visible. Lazy-loading tab data on click would cut initial load by 60-70%.
  • select("*") on core tables: crm_contacts, crm_activities, and crm_emails all use select("*"). Activities include full descriptions, emails include HTML bodies. Selecting only display columns would reduce payload by 40-60%.

Global Architecture

These issues affect every page in the portal and compound with the area-specific problems above.

  • Middleware + layout double query: Middleware queries the profiles table to check roles. Then the authenticated layout queries profiles again via session-cache.ts. The fix is to store role data in a session cookie or custom header after middleware validates, so the layout can skip the redundant query. Estimated savings: 200-300ms per page load.
  • New Supabase client per hook: useDealMessages, useNotifications, and useActivityTracker each call createClient() directly, creating a new instance with independent auth refresh logic. A singleton pattern or Context-injected client would prevent this.
  • 6 nested context providers: The authenticated layout wraps children in ImpersonationProvider, ViewAsProvider, ModuleAccessProvider, MobileLayoutWrapper, ActivityTrackerProvider, and ModuleGuard. Without React.memo on these providers, any state change cascades re-renders through the entire component tree.
  • Polling without visibility checks: useDealMessages (10s), SecureUploadClient (60s), and borrower messaging (15s) all poll on fixed intervals without checking document.visibilityState. When tabs are hidden, these polls continue burning API calls for nothing.
